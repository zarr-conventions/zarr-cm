# Track upstream v0.1 (spatial r3, proj r3, multiscales package) — Design

**Date:** 2026-06-16 **Branch:** `bump-proj-and-spatial` **Scope:** Snapshot the
newly-published upstream `v0.1` of spatial, proj, and multiscales as new package
revisions; convert multiscales to a revision package; wire multiscales into the
drift checker; refresh examples.

## Motivation

Upstream published real `v0.1` git tags for all three tracked conventions and
moved their schema URLs from the dangling `refs/tags/v1` (which never existed as
a tag) to `refs/tags/v0.1`. Our vendored snapshots and `SCHEMA_URL` constants
point at the dead `v1`, so:

- `detect()` keys on the `v1` URL and fails to recognize real-world `v0.1`
  documents.
- The vendored `v1` URL fails the upstream schema's `schema_url` `const` check.

An audit (2026-06-16) found:

| Convention      | UUID      | Change vs our current snapshot                                                                                                                    | Kind        |
| --------------- | --------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ----------- |
| **spatial**     | unchanged | `v0.1` tag; const URLs `v1`→`v0.1`. Field shapes **byte-identical** to our r2.                                                                    | URL only    |
| **proj**        | unchanged | `v0.1` tag; URLs `v1`→`v0.1`. **AND** `proj:code` pattern `^[A-Z]+:[0-9]+$`→`^[^:]+:[^:]+$`; CRS rule `oneOf`→`anyOf`.                            | Substantive |
| **multiscales** | unchanged | `v0.1` tag; URLs `v1`→`v0.1`. Field shape **unchanged**. (A `release-1.0.0` branch — unreleased — adds a required `version` field; out of scope.) | URL only    |

This is the "upstream drift → snapshot a new revision" path the revision
machinery exists for. We snapshot `v0.1` as new revisions (never editing shipped
revisions), and bring multiscales — which never got the revision-package
treatment — to parity with spatial/proj.

## Revision identities (commit-pinned)

Each new revision pins `SCHEMA_URL`/`SPEC_URL` to the `v0.1` tag's commit SHA
(our durable-identity convention; upstream pins to the `refs/tags/v0.1` tag ref,
but a commit SHA cannot move):

- **spatial r3** → `54d81b7ced0376e63ee10f34db31db7d08dcc28d`
- **proj r3** → `5ca5b2f92e5c7245f957d9128b289ee535f0720d`
- **multiscales r2** → `9b78efa75fef0fed302d9cf880037c569354d860`

URL form (example, spatial):
`https://raw.githubusercontent.com/zarr-conventions/spatial/54d81b7ced0376e63ee10f34db31db7d08dcc28d/schema.json`
and the matching `/blob/<sha>/README.md` spec URL.

Latest after this work: **spatial=r3, proj=r3, multiscales=r2**. license/uom
stay flat (`detect()` → `"v1"`). Revision labels remain per-convention opaque
strings keyed off `schema_url`; no cross-convention coordination.

## 1. spatial: add r3

`src/zarr_cm/spatial/`:

- **New `_r3.py`** — a copy of `_r2.py` (same strict-2D validation: dimensions
  exactly 2, bbox exactly 4, transform exactly 6, shape exactly 2 with items ≥1;
  registration enum; same `SpatialAttrs`/`SpatialConventionAttrs`; same shared
  UUID `689b58e2-…`) with `_COMMIT = "54d81b7…"` and the v0.1-pinned
  `SCHEMA_URL`/`SPEC_URL`/`CMO`. `_r2.py` is **not modified** (frozen snapshot).
- **`__init__.py`** — `_REVISIONS = {"r1": _r1, "r2": _r2, "r3": _r3}`;
  `LATEST = "r3"`; add public `r3 = _r3`; switch the package-level `X`
  re-exports (`CMO`, `CONVENTION_KEYS`, `SCHEMA_URL`, `SPEC_URL`, `UUID`,
  `SpatialAttrs`, `SpatialConventionAttrs`) to import from `._r3`; add `"r3"` to
  `__all__` (sorted). `_SCHEMA_URL_BY_REVISION` rebuilds from `_REVISIONS`
  automatically.

## 2. proj: add r3 (relaxed validation)

`src/zarr_cm/proj/`:

- **New `_r3.py`** — a copy of `_r2.py` with these deltas (and `_r2.py`
  untouched):
  - `_COMMIT = "5ca5b2f…"`; v0.1-pinned `SCHEMA_URL`/`SPEC_URL`/`CMO`.
  - `_CODE_PATTERN = re.compile(r"^[^:]+:[^:]+$")` (was `^[A-Z]+:[0-9]+$`).
  - `validate()` requires **at least one** of `proj:code`/`proj:wkt2`/
    `proj:projjson` (was exactly one). Concretely:
    `present = [...]; if not present: raise ValueError(...)` — drop the
    `len(present) != 1` check, keep the `proj:code` pattern check.
  - Same TypedDicts, same shared UUID `f17cb550-…`.
- **`__init__.py`** — register `"r3": _r3`, `LATEST = "r3"`, `r3 = _r3`,
  re-exports → `._r3`, `"r3"` in `__all__`.

## 3. multiscales: flat module → revision package

Convert `src/zarr_cm/multiscales.py` into the package `src/zarr_cm/multiscales/`
mirroring how spatial/proj were first packaged:

- `git mv src/zarr_cm/multiscales.py src/zarr_cm/multiscales/_r1.py` —
  **verbatim**, keeping its current (dangling `v1`) URLs so any document already
  written with them round-trips to r1. Remove the `detect` /
  `_SCHEMA_URL_BY_REVISION` that were added to the flat module earlier (they
  move into the facade — see below) **or** keep them in `_r1.py` harmlessly; the
  facade's `detect` is what the public API uses. Decision: strip the flat
  `detect`/`_SCHEMA_URL_BY_REVISION` from `_r1.py` so each revision module stays
  a pure data/validation snapshot (matching `spatial/_r1.py`, which has no
  `detect`).
- **New `_r2.py`** — copy of `_r1.py`'s shape with `_COMMIT = "9b78efa…"` and
  v0.1-pinned URLs/`CMO`. Same `MultiscalesAttrs`/`LayoutObject`/`Transform`,
  same `layout`-required + `derived_from`⇒`transform` validation, same UUID
  `d35379db-…`.
- **New `__init__.py`** — facade matching `spatial/__init__.py` exactly:
  `from . import _r1, _r2`; `_REVISIONS = {"r1": _r1, "r2": _r2}`;
  `LATEST = "r2"`; `r1`/`r2`; `_SCHEMA_URL_BY_REVISION` from `_REVISIONS`;
  `_resolve_read_revision`; `detect`; `_revision`; dispatch
  `create`/`insert`/`validate`/`extract` with `revision=` (writes default
  LATEST, reads auto-detect — same as spatial/proj); re-export latest types
  (`MultiscalesAttrs`, `LayoutObject`, `Transform`,
  `MultiscalesConventionAttrs`, `CMO`, `CONVENTION_KEYS`, `SCHEMA_URL`,
  `SPEC_URL`, `UUID`); `__all__`.

**Consequences:**

- multiscales gains `revision=` on create/insert/validate/extract (it had none).
- `multiscales.detect()` now returns `"r1"`/`"r2"` instead of `"v1"`.
- **`_contract.py`**: replace
  `_check_multiscales: ConventionModule = _multiscales` with
  `_check_multiscales_r1: ConventionModule = _multiscales_r1` and
  `_check_multiscales_r2: ConventionModule = _multiscales_r2` (importing the
  submodules under `TYPE_CHECKING`), exactly the migration the contract spec
  anticipated.

## 4. Top-level package

`src/zarr_cm/__init__.py`:

- `from . import multiscales` still resolves (now a package); `_REGISTRY`
  unchanged (value is the `multiscales` package, which now exposes `UUID`,
  `detect`, `_REVISIONS` via its facade). `_detect_conventions`,
  `detect_revisions`, `_rev_kwargs`/`_read_rev_kwargs` keep working; multiscales
  now participates in the `revisions={"multiscales": ...}` override paths (a
  capability gain, consistent with spatial/proj).
- `ALL_CONVENTION_KEYS` and `MultiConventionAttrs` reference `multiscales.*` —
  values unchanged, resolved via the package re-exports. Confirm the
  `from .multiscales import LayoutObject, MultiscalesAttrs, MultiscalesConventionAttrs, Transform`
  import still resolves from the package.

## 5. Vendored snapshots + drift checker

- Vendor three new snapshot files (the `v0.1` schemas) at:
  - `.github/upstream-schemas/spatial/54d81b7ced0376e63ee10f34db31db7d08dcc28d.json`
  - `.github/upstream-schemas/proj/5ca5b2f92e5c7245f957d9128b289ee535f0720d.json`
  - `.github/upstream-schemas/multiscales/9b78efa75fef0fed302d9cf880037c569354d860.json`
    (The old `v1`-era snapshots stay as the r2/r1 fixtures.)
- `.github/scripts/check_upstream.py` `TRACKED`: add a `multiscales` entry, and
  retarget all three `vendored` paths to the new v0.1 snapshot files (the script
  compares upstream `main` to the _newest_ vendored snapshot). After this,
  running the checker reports **OK** for all three.
- `tests/schemas/`: add `spatial-r3.json`, `proj-r3.json`, `multiscales-r2.json`
  (copies of the new vendored snapshots) for the per-revision fixture tests.

## 6. Tests

- **spatial**: add r3 tests in `tests/test_spatial.py` — r3 validates 2D,
  rejects 3D (same as r2), `SCHEMA_URL` contains the v0.1 SHA and not
  `refs/tags/v1`; r3 `create`→`insert` validates against
  `tests/schemas/spatial-r3.json`.
- **proj**: add r3 tests in `tests/test_proj.py` — r3 ACCEPTS a code r2 rejects
  (e.g. `"urn:ogc:def:crs:EPSG::4326"` or lowercase authority) proving the
  relaxed pattern; r3 accepts **two** CRS keys together (anyOf) while r2 still
  rejects that (oneOf); r3 still rejects zero keys; r3 `create`→`insert`
  validates against `tests/schemas/proj-r3.json`; r3 `SCHEMA_URL` is
  v0.1-pinned.
- **multiscales**: new/updated tests mirroring `test_spatial.py` for the package
  — r1/r2 round-trip, r2 fixture validation, `detect` returns `r1`/`r2`,
  `revision=` override works. Update existing `tests/test_multiscales.py` to the
  package API (it currently imports the flat module; the public
  `zarr_cm.multiscales` API is unchanged for the no-revision calls, so most
  tests pass as-is — adjust any that asserted the old `v1` URL/CMO).
- **detect**: update `tests/test_detect.py` — multiscales `detect` now returns
  `"r1"`/`"r2"` (not `"v1"`); the flat-detect test for multiscales moves to the
  revisioned-style assertions; `detect_revisions` aggregate for a latest
  multiscales doc returns `"r2"`.
- **contract**: `tests/test_contract.py` enumerates the live dispatch set, so it
  picks up multiscales' new `_REVISIONS` automatically (now spatial r1/r2/r3,
  proj r1/r2/r3, multiscales r1/r2, license, uom = 10 targets). The existing
  `>= 7` sanity assert still holds (10 ≥ 7) — no edit needed unless it was an
  exact `==`; confirm and leave as-is if `>=`.
- **examples test**: `tests/test_examples.py` runs the scripts as-is.

## 7. Examples

Update narration/behavior to the new latest:

- `examples/spatial.py` — latest is now r3 (2D); migration still shows the lossy
  r1-3D case. Update any printed "r2" expectations to "r3".
- `examples/proj.py` — latest r3; the migrate workflow now additionally
  demonstrates that r3 accepts a code r2 rejected (relaxed pattern), making the
  read/migrate story richer.
- `examples/multiscales.py` — now a revisioned convention: `detect` returns
  `r1`/`r2`, `revision=` available; the migrate workflow becomes a real r1→r2
  migration (same shape, new v0.1 identity) instead of the identity scaffold.
- `examples/license.py`, `examples/uom.py` — unchanged (still flat, `"v1"`).

## Out of scope

- multiscales `release-1.0.0` / the required `version` field (unreleased branch;
  a future r3 when `v1.0.0` ships).
- Adopting upstream's tag-ref URL form (we keep commit-SHA pinning).
- Changing the revision-label scheme to encode upstream versions (labels stay
  per-convention opaque).
- license/uom (no upstream change; still flat).
